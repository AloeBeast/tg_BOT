# Шаг 1: Миграция SQLite → PostgreSQL (asyncpg + пул соединений)

## Цель
Заменить синхронный SQLite на асинхронный PostgreSQL с пулом соединений. Фундамент для масштабирования на 1k–100k пользователей и будущих подписок.

## Решения архитектуры

- **Пул соединений** — глобальный синглтон в `database.py`: `init_pool()` / `close_pool()` / `get_pool()`. Минимальный diff в хендлерах (только `await`), без смены сигнатур.
- **Сырой asyncpg** без ORM. `database.py` уже репозиторий из 6 функций — сохраняем форму.
- **DDL через `IF NOT EXISTS`** на каждом старте. Миграционный фреймворк (Alembic) — позже, когда появятся подписки.
- **Все DB-функции → `async def`**. Хендлеры обновляются до `await`.

## Схема PostgreSQL

```sql
CREATE TABLE IF NOT EXISTS users (
    user_id    BIGINT PRIMARY KEY,        -- BIGINT: Telegram ID > INT32
    name       TEXT NOT NULL DEFAULT '',
    facts      TEXT NOT NULL DEFAULT '',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS messages (
    id         BIGSERIAL PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    role       TEXT NOT NULL,
    content    TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_messages_user_id_id
    ON messages (user_id, id DESC);
```

Ключевые отличия от SQLite-схемы:
- `user_id` → `BIGINT` (критично: Telegram ID переполняет INT4)
- `id` → `BIGSERIAL`
- `timestamp` → `created_at TIMESTAMPTZ`
- `updated_at` добавлен в `users` для будущих подписок
- Без FK `messages → users` (порядок вызовов в хендлерах не гарантирует user-row-first)

### 4. `database.py` — полная переработка
- `import asyncpg` вместо `import sqlite3`
- Глобальный `_pool: asyncpg.Pool | None`
- `init_pool()` → `await asyncpg.create_pool(dsn=DATABASE_URL, min_size, max_size, command_timeout=30)`
- `close_pool()` → `await _pool.close()`
- `get_pool()` → возвращает пул или `RuntimeError`
- `init_db()` → async, DDL через `pool.execute()` с `IF NOT EXISTS` + создание индекса
- `save_message()` → `await pool.execute("INSERT ... VALUES ($1, $2, $3)", ...)`
- `get_history()` → `await pool.fetch(...)` + `reversed(rows)` → list of dicts
- `get_user_profile()` → `fetchrow`, если None — INSERT + вернуть пустой профиль
- `update_user_name()` → `INSERT ... ON CONFLICT DO UPDATE SET name = EXCLUDED.name, updated_at = NOW()`
- `append_user_fact()` → **транзакция с `FOR UPDATE`**: upsert через `ON CONFLICT DO NOTHING`, затем `SELECT ... FOR UPDATE`, read-modify-write в Python, `UPDATE`. Устраняет race condition.

Плейсхолдеры: `?` → `$1, $2, ...` во всех запросах.

### 5. `main.py` — lifecycle пула
- Импорт `init_pool, close_pool`
- `await init_pool()` перед `init_db()`
- `await init_db()` (теперь async)
- `try/finally`: `close_pool()` в `finally` для graceful shutdown

## Риски и гоччи

1. **`BIGINT` везде** — Telegram user_id переполняет INT4 (2.1 млрд). asyncpg строго типизирован.
2. **Плейсхолдеры `$1, $2`** — не `?`. Пропуск = runtime error.
3. **asyncpg `Record`** — доступ по ключу `r["role"]`, не tuple unpacking.
4. **`create_pool` внутри event loop** — вызывать в `main()`, не при импорте модуля.
5. **Race в `append_user_fact`** — решён через `FOR UPDATE` транзакцию.
6. **`max_connections` Postgres** — дефолт 100. При нескольких процессах бота: `Σ max_size ≤ ~80`.
7. **Пароль с спецсимволами** в `DATABASE_URL` — URL-encode.
